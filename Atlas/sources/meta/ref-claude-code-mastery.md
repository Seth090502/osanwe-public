# Claude Code Power-User Reference

## 1. Architecture and surface taxonomy

Claude Code is a **single agent harness with many fronts**. Anthropic's overview describes it as "an agentic coding tool that reads your codebase, edits files, runs commands, and integrates with your development tools" exposed as a terminal CLI, an IDE extension family (VS Code, JetBrains plugin ID 27310, Cursor), a desktop application (macOS Intel/Apple Silicon, Windows x64, Windows ARM64), a web app at `claude.ai/code` mirrored in the iOS Claude app, a Slack bot, a Chrome extension (beta), the Computer Use preview, and a GitHub Action (`anthropics/claude-code-base-action`). Every surface terminates at the same engine, so CLAUDE.md, settings, MCP servers, hooks, and skills are portable across them[^1][^2]. The **Agent SDK** -- formerly the "Claude Code SDK", renamed in late 2025 -- ships as `@anthropic-ai/claude-agent-sdk` (TypeScript) and `claude_agent_sdk` (Python) and reuses the same harness; setting `setting_sources=["project"]` opts the SDK into project-level CLAUDE.md, skills, hooks, and `.mcp.json`[^3].

Claude Code exposes **six distinct extension primitives** that operators frequently conflate. **Skills** are `SKILL.md`-keyed directories under `.claude/skills/<name>/` (project), `~/.claude/skills/<name>/` (user), or `<plugin>/skills/<name>/` (plugin). Skills are loaded only when invoked, either by user (`/<name>`) or model (description-triggered)[^4]. **Subagents** are markdown files in `.claude/agents/` invoked through the **Agent tool** (the renamed `Task` tool, since Claude Code v2.1.63), which spawns a fresh context window with its own tool list and returns only the final assistant message to the parent[^5]. **Slash commands** include the bundled set described in sec. 2 plus any project commands, skills (legacy `.claude/commands/*.md` is now superseded by `.claude/skills/<n>/SKILL.md`; if both exist the skill wins), and MCP prompts (`/mcp__<server>__<prompt>`)[^4]. **Hooks** are user-defined shell, HTTP, prompt-LLM, or sub-agent handlers wired to ~25 lifecycle events. **MCP tools** appear as standard tools named `mcp__<server>__<tool>` with lazy loading via the Tool Search mechanism shipped early 2026 (~95% context savings on large MCP fleets). The **underlying API** -- Claude Opus 4.7, Sonnet 4.6, Haiku 4.5 as of April 2026 -- is the only thing actually billed; everything else just modifies what the model sees or can do.

**Permission modes** number six as of April 2026, not four[^6]. Shift+Tab cycles `default -> acceptEdits -> plan -> auto` in interactive mode. `default` prompts on first use of each tool; macOS sandbox-exec wraps read-only Bash automatically. **`acceptEdits`** auto-approves filesystem edits and the safe filesystem Bash subset (`mkdir`, `touch`, `rm`, `mv`, `cp`, `sed`) but still prompts on other Bash. **`plan`** allows reads and shell research but blocks edits; `ExitPlanMode` currently resets to `acceptEdits` regardless of prior mode (issue #39973). **`auto`** (released March 2026 and extended to Max users alongside Opus 4.7) adds a Sonnet 4.6 classifier that auto-approves safe actions and blocks scope escalation. **`dontAsk`** silently denies anything outside `allowed_tools` -- designed for CI. **`bypassPermissions`** equals `claude --dangerously-skip-permissions`; deny rules and hooks still execute before the mode check, and admins can disable it globally via managed `permissions.disableBypassPermissionsMode: "disable"`. Set persistently with `{"permissions": {"defaultMode": "acceptEdits"}}` in any `settings.json`.

**Configuration hierarchy** has six precedence layers (highest -> lowest)[^7]: enterprise managed policy (`/Library/Application Support/ClaudeCode/managed-settings.json` on macOS, `/etc/claude-code/managed-settings.json` on Linux), `managed-settings.d/` drop-ins (added v2.1.83), `.claude/settings.local.json` (auto-gitignored), `.claude/settings.json` (project-checked-in), `~/.claude/settings.json` (user-global), then CLI flags and env vars. **Memory hierarchy** (CLAUDE.md) loads lowest-to-highest priority because the model attends more strongly to recent context: auto-memory output -> deprecated `CLAUDE.local.md` -> `~/.claude/CLAUDE.md` and `~/.claude/rules/*.md` -> project `.claude/rules/*.md` -> project-root `CLAUDE.md` -> managed drop-ins -> managed policy CLAUDE.md (which `claudeMdExcludes` cannot suppress). The **`@import` mechanism** (`@./docs/spec.md`, `@~/.claude/preferences.md`) recurses up to **depth 5**, ignores imports inside fenced code, silently skips circular references and missing files, prompts once for paths outside the project tree, and **does not save tokens** because imports load eagerly at session start. Project-root CLAUDE.md is re-read from disk after `/compact`; nested subdirectory CLAUDE.md files are not re-injected automatically.

## 2. Built-in skills and slash commands

Claude Code ships **60+ built-in slash commands and 5 bundled skills** as of April 2026[^4][^8]. The full menu populates when the user types `/` in interactive mode. The **project and memory commands** include `/init` (analyzes the codebase and writes a starter CLAUDE.md; pitfall: in `~/.claude/` it overwrites an existing CLAUDE.md without prompting per issue #21795, and the welcome screen still nags about `/init` even when `.claude/CLAUDE.md` exists per issue #26225); `/memory` (opens the editor and toggles **Auto Dream** consolidation, the feature that the brief named `consolidate-memory` -- it is not a standalone skill but a `/memory`-toggled background pass that orients, gathers, consolidates, and prunes `~/.claude/projects/<project>/memory/`); and `/team-onboarding` (added v2.1.101, April 11 2026, generates a teammate ramp-up doc from CLAUDE.md, skills, subagents, and hooks).

The **review and quality commands** are `/review` (single-pass local code review whose output quality varies sharply with the active `/model` and `/effort` setting; pin Opus 4.7 at `xhigh` effort or chain with `/simplify` for stable triangulation), `/security-review` (diff-only scan covering injection, auth, secrets, crypto, mirrored exactly by the `anthropics/claude-code-security-review` GitHub Action; false positives spike on multi-commit PRs unless `max-commits` is configured)[^9], `/ultrareview` (detailed below), and `/insights` (past-month HTML usage report).

**Session and context commands** include `/clear`, `/compact [focus instructions]`, `/context` (visualizes context-window allocation as a colored grid), `/recap` (away-summary, toggle via `CLAUDE_CODE_ENABLE_AWAY_SUMMARY`), `/usage` (the merger of the legacy `/cost` and `/stats`; both names still work), `/resume`, `/fork`, `/export`, and `/btw <q>` for sidecar questions that do not pollute the main thread. **Configuration commands** include `/config`, `/permissions`, `/less-permission-prompts` (session-scoped allowlist), `/sandbox`, `/model` (warns before mid-session swap because the next response re-reads history uncached), `/effort [low|medium|high|xhigh|max]`, and `/theme`. **Extension management** runs through `/agents`, `/hooks`, `/skills`, `/mcp`, `/plugin`, `/plugin marketplace add`, and `/reload-plugins`. **Auth and diagnostics** include `/login` (env `CLAUDE_CODE_OAUTH_TOKEN` overrides disk credentials), `/logout`, `/doctor`, `/bug`, `/help`, `/vim` (with v/V visual selection in latest builds), `/terminal-setup`, and `/install-github-app`. **Background and tasks** are managed through `/tasks`, `/bashes`, `/loop [interval] [prompt]` (alias `/proactive`), `/schedule` (Routines on Anthropic infrastructure), and `/pr_comments`.

The **bundled skills** listed at code.claude.com/docs/en/skills are `/simplify`, `/batch`, `/debug`, `/loop`, and `/claude-api`[^4]. `/simplify` runs a three-agent pipeline (architectural review, duplicate-logic, performance) intended as a pre-PR quality gate, announced by Boris Cherny on Threads[^10]. `/batch` fans parallelizable migrations across 5-30 isolated git worktrees, runs tests after edits, and opens PRs. `/debug` reads the Claude Code debug log to triage failed tool calls. `/loop` is the self-paced or interval-based recurring task discussed in sec. 8. `/claude-api` loads the Claude API plus Agent SDK plus Managed Agents reference scoped to the project's detected language and supports `/claude-api migrate <scope> to claude-opus-4-7` for cross-codebase model migrations[^11].

The **`anthropics/skills` repository** at github.com/anthropics/skills (124k stars as of April 2026) doubles as the official plugin marketplace. The brief's "anthropic-skills:*" namespace does not exist as a top-level prefix; instead, after `/plugin marketplace add anthropics/skills`, skills install under `document-skills@anthropic-agent-skills` (containing `docx`, `pdf`, `pptx`, `xlsx`) and `example-skills@anthropic-agent-skills`. Plugin skills surface as `<plugin>:<skill>` (e.g., `document-skills:pdf`), preventing collisions. The `setup-cowork` skill the brief mentions belongs to Claude Cowork rather than Claude Code itself, and `schedule` lives as the `/schedule` Routines command and `/loop`. **`skill-creator`** is the meta-skill that walks operators through SKILL.md frontmatter, runs paired with-skill/without-skill A/B evaluations, and auto-generates a `.skill` archive -- its evaluation harness is the canonical reference pattern documented in sec. 7. **`user-invocable: false`** hides a skill from the slash menu but still exposes its description for model-side invocation; **`disable-model-invocation: true`** is stronger and prevents the description from loading into context at all.

**`/ultrareview` deserves its own paragraph**[^12]. Available in Claude Code v2.1.86+ and shipped publicly in v2.1.111 alongside Opus 4.7, it launches a **fleet of independent reviewer agents in a remote Claude Code on the web sandbox**; every finding is "independently reproduced and verified" before it surfaces. **Authentication requires Claude.ai login** (run `/login` first if API-key authenticated), and it is **unavailable on Amazon Bedrock, Google Vertex AI, Microsoft Foundry, and Zero-Data-Retention organizations**. Branch mode (`/ultrareview`) reviews diff between current branch and default branch, including uncommitted changes, by bundling the working tree to the sandbox; PR mode (`/ultrareview 1234`) clones the PR directly from github.com and is required when the working tree exceeds bundle limits. Pro and Max plans get **3 free runs that expire May 5 2026 and do not refresh**; Team and Enterprise get zero free runs. Beyond free runs, every invocation bills against extra usage at **$5-$20 per run** depending on change size; the launch dialog shows the estimate, and the account must have `/extra-usage` enabled. Runtime is typically **10-20 minutes** as a background task, terminal closure is safe, and progress is tracked through `/tasks`. Worth invoking pre-merge on substantive auth, data, or integration changes; not worth it for fast iterative feedback where `/review` returns in seconds.

## 3. Subagents (the Agent tool ecosystem)

The harness ships a **small first-party set of subagents** that operators systematically under-use[^5][^13]. **Explore** runs on Haiku with read-only tools and accepts a `thoroughness` parameter (`quick`, `medium`, `very thorough`); it is the right call for codebase search and file discovery and the wrong call for modification or multi-step reasoning. **Plan** inherits the main model, restricts itself to read-only tools, and prevents infinite nesting (subagents cannot spawn subagents). **general-purpose** inherits the main model and has the full tool surface; it is the default target when fork mode is enabled via `CLAUDE_CODE_FORK_SUBAGENT=1`. **statusline-setup** runs on Sonnet and is invoked when `/statusline` configures the status line. **claude-code-guide** runs on Haiku for Q&A about Claude Code itself. **security-review** ships as a slash command backed by a specialized prompt sourced from `anthropics/claude-code-security-review` and excludes DOS, theoretical races, lack-of-hardening, secrets-on-disk, and SSRF that only controls path. **code-reviewer** is not built-in by default but is the canonical example in the docs (read-only `Read, Grep, Glob, Bash`).

**Authoring custom subagents** uses YAML frontmatter where only `name` and `description` are required[^5]:

```yaml
---
name: code-reviewer
description: Reviews code for correctness, style, and security
tools: Read, Glob, Grep              # allowlist; omit to inherit all
disallowedTools: Write, Edit         # denylist; applied first
model: sonnet                        # sonnet|opus|haiku|inherit|<full-id>
permissionMode: default
maxTurns: 25
skills: [api-conventions, error-handling-patterns]
mcpServers: [github, { playwright: { type: stdio, command: npx, args: ["-y","@playwright/mcp@latest"] } }]
hooks: { PreToolUse: [{ matcher: "Bash", hooks: [{ type: command, command: ./scripts/check.sh }] }] }
memory: project                      # user|project|local; ~/.claude/agent-memory/<name>/
background: false                    # true = always background
effort: high
isolation: worktree                  # spawn in temp git worktree, auto-cleanup if no changes
color: cyan
initialPrompt: "..."
---
```

**Model resolution** flows `CLAUDE_CODE_SUBAGENT_MODEL` env -> per-invocation parameter -> frontmatter `model` -> main conversation model. **Tool restriction** applies `disallowedTools` first, then resolves `tools` against the remaining pool. **Plugin-loaded subagents intentionally ignore `hooks`, `mcpServers`, and `permissionMode`** for security. To restrict which subagents a parent can spawn: `tools: Agent(worker, researcher), Read, Bash` allowlists specific subagent types; bare `Agent` permits all; omitting `Agent` forbids spawning. To disable specific built-ins globally: `permissions.deny: ["Agent(Explore)"]`.

**Isolation parameter `isolation: "worktree"`** spawns the subagent in a temporary git worktree that auto-cleans if the subagent makes no changes and triggers `WorktreeCreate`/`WorktreeRemove` hooks for non-git VCS bootstrap. **Issue #33045 (April 2026) is critical**: when a *team agent* is spawned through the Agent tool with `isolation: "worktree"`, the parameter is silently ignored and the agent runs in the main repo. Workaround until fixed: pre-create the worktree manually before spawning.

**Foreground vs background semantics** are governed by `background: true` in frontmatter, **Ctrl+B** at runtime, or the natural-language "run this in the background." Before launching a background subagent, Claude Code pre-prompts for any tool permissions; once running, the subagent inherits these and **auto-denies anything not pre-approved**. Foreground subagents block the main conversation and pass permission prompts and `AskUserQuestion` calls through to the user. Background can be globally disabled with `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1`. Note that **Task-tool `run_in_background` for spawning background subagents is feature-requested (issue #9905) but not yet shipped**; today's "background subagent" is realized through Bash `run_in_background` + `claude -p` in tmux, or the `background: true` frontmatter on a custom agent.

**Parallel invocation**: calling the Agent tool multiple times in a single assistant turn is the canonical pattern, with up to ~10 concurrent subagents in a REPL before queuing -- Anthropic engineers typically use 3-5. Subagents start with **fresh context** consisting of their system prompt and basic env (cwd) only; the only inbound channel from parent to subagent is the prompt string. The parent receives the subagent's **final assistant message verbatim** as the Agent tool result; intermediate tool calls remain in the subagent's transcript at `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl`, persist independently of the main conversation, and survive main-conversation compaction. Subagent transcripts auto-clean per `cleanupPeriodDays` (default 30), and subagent context auto-compacts at ~95% (override `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`).

**Forks** (experimental, gated by `CLAUDE_CODE_FORK_SUBAGENT=1` since v2.1.117) inherit the *full* parent conversation, system prompt, tools, and prompt cache; tool calls remain isolated; the final result returns to the main thread. Fork mode forces background spawn for all subagents and replaces general-purpose with forks. **`/agents`** opens a tabbed UI with Running and Library tabs; the `--agent <name>` CLI flag (or `agent` in settings) makes a subagent the main thread for the entire session, replacing the default Claude Code system prompt.

## 4. Hooks system end-to-end

Hooks live in any settings layer (`~/.claude/settings.json`, `.claude/settings.json`, `.claude/settings.local.json`, managed policy, plugin `hooks/hooks.json`, or skill/agent frontmatter), browse via the read-only `/hooks` UI, and disable wholesale with `"disableAllHooks": true`[^14][^15]. Configuration uses tool-name regex matchers and supports four handler types: **`command`** (shell, stdin/stdout/stderr/exit-code), **`http`** (POST event JSON to URL with `headers` and `allowedEnvVars`), **`prompt`** (single-turn LLM, returns `{"ok":bool,"reason":str}`, defaults to Haiku), and **`agent`** (multi-turn subagent verifier, default 60 s timeout, up to 50 turns). Default per-hook timeout is 10 minutes; **`async: true`** (added January 2026) runs the hook in background and forfeits any ability to block.

The **full event taxonomy** as of April 2026 is materially larger than the brief's nine[^14]. Lifecycle events are `SessionStart` (matchers: `startup|resume|clear|compact`), `SessionEnd` (matchers: `clear|resume|logout|prompt_input_exit|bypass_permissions_disabled|other`), `UserPromptSubmit`, `UserPromptExpansion`, `PreToolUse`, `PermissionRequest` (does not fire in `-p` mode), `PermissionDenied` (return `{retry:true}` to permit retry after auto-mode classifier denies), `PostToolUse`, `PostToolUseFailure`, `PostToolBatch` (after a parallel batch resolves, before next model call), `Notification` (matchers: `permission_prompt|idle_prompt|auth_success|elicitation_dialog`), `SubagentStart`, `SubagentStop` (gets `last_assistant_message`), `TaskCreated`, `TaskCompleted`, `Stop` (fires per response, not per task; gets `last_assistant_message`), `StopFailure` (matchers include `rate_limit|auth|billing`), `TeammateIdle`, `InstructionsLoaded` (matchers: `session_start|nested_traversal|path_glob_match|include|compact`), `ConfigChange`, `CwdChanged`, `FileChanged` (matcher is pipe-separated literals, not regex), `WorktreeCreate`, `WorktreeRemove`, `PreCompact` (matchers: `manual|auto`), `PostCompact`, `Elicitation`, `ElicitationResult`, and `Setup` (`--init`, `--init-only`, `--maintenance` CLI flags, v2.1.10+, has access to `CLAUDE_ENV_FILE`).

**Configuration shape** with the v2.1.85+ `if` field (permission-rule syntax for fine-grained matching including compound commands):

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/protect-files.sh",
        "timeout": 30,
        "if": "Edit(*.ts)",
        "async": false
      }]
    }]
  }
}
```

**Output semantics**: exit code `0` proceeds, and for `UserPromptSubmit`, `UserPromptExpansion`, and `SessionStart` stdout is **appended to Claude's context** (silent for other events). Exit code `2` blocks and feeds stderr back to Claude as feedback. Any other code proceeds with a warning. Structured JSON to stdout uses `hookSpecificOutput.permissionDecision: "allow|deny|ask|defer"`, optional `updatedInput` to rewrite the tool call (last writer wins on parallel hooks), and `additionalContext` to inject a system-reminder string. **Mixing exit 2 with JSON loses the JSON** -- exit 2 wins. **Hook `"allow"` does not override `permissions.deny` rules**, especially managed policy denies; hook `"deny"` blocks even in `bypassPermissions` and `--dangerously-skip-permissions`. When multiple hooks match, all run in parallel, identical commands are deduplicated, decisions resolve to the most restrictive (`deny > ask > allow`), and `additionalContext` strings concatenate.

**Master community patterns** worth importing: **path-guard** PreToolUse hooks read `tool_input.file_path`, check against `.env`, `package-lock.json`, and `.git/`, and exit 2 with reason. **Prompt log** UserPromptSubmit pipes `jq -r .prompt >> ~/.claude/prompt-log/$(date +%Y-%m-%d).md`. **Auto-PR-on-Stop** runs `gh pr create` after task completion with a `stop_hook_active` recursion guard. **Hot-cache load on SessionStart** echoes `git status --short`, recent issues, and env vars to stdout for context injection; with the `compact` matcher it re-injects after compaction. **Direnv integration** combines `direnv export bash > "$CLAUDE_ENV_FILE"` on `Setup`, `CwdChanged`, and `FileChanged: .envrc|.env`. **Auto-format-and-commit on PostToolUse** with `matcher: "Edit|Write|MultiEdit"` runs prettier or black on the touched file. **PreCompact transcript backup** copies the session JSONL to `.claude/backups/`. **TDD enforcement** uses an `agent`-typed Stop hook that runs the test suite and returns `{ok:false, reason:"tests failing"}` to force continuation.

**Failure modes** to internalize: **silent hooks** are the default -- successful runs produce no UI output, so use `Ctrl+O` for verbose transcript or `claude --debug-file /tmp/claude.log`. **Recursive triggers** kill sessions: a UserPromptSubmit hook that spawns subagents can infinite-loop, and a Stop hook that returns exit 2 unconditionally creates a stop-resume loop -- **always check `stop_hook_active`**. **Context overflow** happens when stdout from UserPromptSubmit or SessionStart gets injected verbatim; verbose `git log` or `find .` output silently consumes the window. **Shell profile pollution** breaks JSON parsing when `~/.zshrc` unconditionally echoes; wrap in `if [[ $- == *i* ]]; then ... fi`. **Non-determinism** between parallel hooks setting `updatedInput` -- last finisher wins. **HTTP hook errors are non-blocking** by default; non-2xx responses, connection failures, and timeouts continue execution unless the hook returns 2xx with `permissionDecision: "deny"`. **Plugin-loaded hooks are ignored** entirely on plugin agents/skills. **`-p` non-interactive mode skips PermissionRequest** -- use PreToolUse instead.

## 5. MCP server ecosystem

The **Model Context Protocol** is an open standard Anthropic announced 2024-11-25 with a one-year-anniversary spec re-cut released 2025-11-25[^16]. The wire format is JSON-RPC 2.0 over stateful sessions; the architecture pairs a host application (Claude Code) running one or more clients with **1:1 server connections** negotiated at `initialize` time. **Server primitives** are tools (model-invocable, "POST-like", with optional `outputSchema`), resources (read-only, surfaced via `@server:protocol://path`, "GET-like"), and prompts (templates appearing as `/mcp__server__prompt`). **Client primitives** are sampling, elicitation, roots, logging, progress, and cancellation. **Transports** are stdio (newline-delimited JSON-RPC over stdin/stdout, child process), Streamable HTTP (HTTP POST plus optional SSE, supports resumable streams via `Last-Event-ID` and stateful `Mcp-Session-Id`), and the deprecated SSE-only transport from protocol version 2024-11-05.

**Top servers by use case** as of April 2026[^17][^18]: developer productivity covers `@modelcontextprotocol/server-filesystem`, `mcp-server-git` (uvx), the **GitHub MCP server** at `ghcr.io/github/github-mcp-server` or remote `https://api.githubcopilot.com/mcp` (the npm `@modelcontextprotocol/server-github` was deprecated April 2025), Postgres via `@bytebase/dbhub` or community servers (the original is archived), and Sentry at `https://mcp.sentry.dev/mcp` over OAuth. Browsing covers Microsoft's **Playwright MCP** (`@playwright/mcp`), which uses accessibility-tree snapshots rather than screenshots -- distinct from the community `@executeautomation/playwright-mcp-server`. Desktop control comes via Anthropic's Computer Use tool plus community servers. Claude Code introspection includes `claude-code-docs` HTTP at `https://code.claude.com/docs/mcp`, the `claude-code-guide`, and `claude mcp serve` (Claude Code itself as an MCP server, letting other agents drive its tools). Data integrations include Notion (`https://mcp.notion.com/mcp`), Linear, Slack (`https://mcp.slack.com/mcp` with scoped OAuth), Gmail, Google Drive, Google Calendar, Stripe, PayPal, HubSpot, Asana, Atlassian, Figma, Canva, and Zapier. Discovery flows through the **Official MCP Registry** (`registry.modelcontextprotocol.io`, launched preview 2025-09-08, ~2,000 servers by November 2025, reverse-DNS namespacing) and **Smithery.ai** (~7,300 servers in early 2026; `smithery mcp search`, `smithery mcp add <url> --client claude`).

**Prefer dedicated MCP over Bash** when typed schemas matter (each tool advertises `inputSchema`, eliminating prose-to-flag translation), when parallel calls help (Claude Code can invoke multiple MCP tools concurrently in one turn versus Bash's serial shell), when structured output is consumed downstream (`structuredContent` validated against `outputSchema`), when shell escaping creates bugs (arguments are JSON, not strings), when permissioning needs granularity (rules can target `mcp__server__tool` patterns), and when output size matters (`_meta.anthropic/maxResultSizeChars` plus automatic disk persistence on overflow).

**Authoring a custom MCP server** is brief in either SDK[^19][^20]. The minimum-viable Python (FastMCP, `mcp` PyPI package, SDK >=1.2):

```python
from mcp.server.fastmcp import FastMCP
mcp = FastMCP("my-server")

@mcp.tool()
def add(a: int, b: int) -> int:
    """Add two numbers"""
    return a + b

@mcp.resource("greeting://{name}")
def greet(name: str) -> str:
    return f"Hello, {name}!"

if __name__ == "__main__":
    mcp.run(transport="stdio")        # or "streamable-http"
```

The TypeScript equivalent uses `@modelcontextprotocol/sdk`'s `McpServer` plus `StdioServerTransport` plus Zod schemas. **SDK choice** comes down to ecosystem: Python wins for data and ML workloads and auto-generates schemas from type hints; TypeScript wins for browser/web stacks and integrates cleanly with Express and Hono middleware. **Registration** in Claude Code uses `.mcp.json` for project scope (prompts approval before first use), `~/.claude.json` for user scope, or claude.ai for connector scope; Claude Desktop uses `claude_desktop_config.json` with the same `mcpServers` schema. Precedence runs local > project > user > plugin > claude.ai connector. **Distribution** is via `npx -y your-pkg`, `uvx your-pkg` or `pipx run your-pkg`, Docker, the Official Registry (`server.json` with namespace ownership via GitHub OAuth or DNS challenge), or Smithery (`smithery mcp publish`).

**Security implications** are non-trivial: stdio servers run as the current user with full filesystem and network privilege, so third-party servers should be treated as untrusted code. **Prompt injection through tool outputs** is the highest-impact risk -- a malicious page fetched by a browsing server can hijack subsequent tool calls. Allowlist via `permissions.allow`/`deny` patterns like `mcp__github__list_issues` or `mcp__postgres__*`; enterprise admins can hard-restrict via `managed-mcp.json` and `allowedMcpServers`/`deniedMcpServers` policies. **OAuth scope pinning** through `oauth.scopes` in `.mcp.json` overrides discovered scopes. **DNS rebinding** is defended by binding local HTTP servers to 127.0.0.1 with `Host` validation (`createMcpExpressApp` does this by default). The **`MAX_MCP_OUTPUT_TOKENS`** environment variable defaults to 25,000 with a warning at 10,000; oversized tool responses persist to disk automatically.

## 6. Memory, context, and compaction

Claude Code runs **two complementary memory mechanisms**[^21]. CLAUDE.md is operator-written, scoped to project/user/org, loads into every session in full, and is meant for conventions, build commands, and architecture. **Auto-memory** is Claude-written, scoped per working tree (per git repo), loads the first 200 lines or 25 KB of `MEMORY.md` at every session, and is meant for learnings and preferences. Auto-memory requires Claude Code v2.1.59+, is on by default, toggles via `/memory` or `"autoMemoryEnabled": false` in settings, and disables entirely with `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`. The directory layout is `~/.claude/projects/<project>/memory/` containing `MEMORY.md` (the auto-loaded index) plus topic files (`debugging.md`, `api-conventions.md`) loaded on demand when Claude reads them with standard file tools. The "Writing memory" / "Recalled memory" footer indicates active reads and writes. **Subagent memory** is separately controlled by the agent's `memory:` frontmatter (`user|project|local`).

**A clarification worth flagging**: the frontmatter schema (`id`, `type`, `tags`, `created`) and the type taxonomy (`user/feedback/project/reference`) some references describe are **not** present in Anthropic's official auto-memory documentation as of April 2026. Anthropic's docs describe MEMORY.md and topic files as plain markdown. Those keys appear in third-party plugins (`claude-mem`, `basic-memory`) and may have been confused with subagent frontmatter (`memory:` field) or rule frontmatter (`paths:` field on `.claude/rules/*.md`). Operators relying on a strict frontmatter schema should standardize one inside their own vault rather than expecting the harness to enforce it.

**The `@import` mechanism** in any CLAUDE.md (`@./docs/spec.md`, `@~/.claude/preferences.md`, or absolute paths) recurses up to depth 5, ignores imports inside fenced code blocks, silently skips circular references and missing files, prompts once before importing paths outside the project tree, and **does not save tokens** because imports load eagerly at session start. Project-root CLAUDE.md is re-read from disk and re-injected after `/compact`, **surviving compaction**; nested subdirectory CLAUDE.md files only reload the next time Claude reads a file in that subdirectory. `--add-dir` does not load CLAUDE.md from added dirs unless `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1`, but `.claude/skills/` always loads regardless.

**Context window management**: default 200K applies to all models; **1M-token context** went GA on Opus 4.6/4.7 and Sonnet 4.6 on March 13 2026[^22]. On Max/Team/Enterprise plans, Opus auto-upgrades to 1M with no flag; disable via `CLAUDE_CODE_DISABLE_1M_CONTEXT=1`. There is **no long-context premium** in the GA release -- a 900K-token request bills at the same per-token rate as a 9K one. Standard pricing is Opus 4.6/4.7 at $5/$25 per MTok in/out and Sonnet 4.6 at $3/$15. The `opusplan` alias runs the plan phase in 200K Opus and executes in Sonnet 4.6; auto-upgrade does not extend to opusplan. Context-aware models receive `<system_warning>Token usage: X/Y; Z remaining</system_warning>` after each tool call.

**Auto-compaction triggers near 95%** of context capacity (community-confirmed; issue #15719 closed as a duplicate confirms the threshold). Behavior: project-root CLAUDE.md survives because it is re-read from disk; nested CLAUDE.md, path-scoped rules, and skill bodies are summarized away and reload on next matching file read or skill invocation; skills are re-injected post-compact but truncated to a per-skill cap with oldest invoked skills dropping first. **`/compact`** supports a focus instruction (`/compact focus on test failures and pending refactors`); custom compaction guidance can also live in CLAUDE.md under a `# Compact instructions` heading. **`/clear`** wipes conversation history entirely. **`PreCompact`** receives `trigger` (`manual`/`auto`) and `custom_instructions`; exit 2 or `{"decision":"block"}` blocks compaction (blocking auto-compaction triggered proactively continues uncompacted; blocking compaction triggered to recover from a context-limit error surfaces the original error). **`PostCompact`** receives `compact_summary` and cannot affect outcome. **`SessionStart` with matcher `compact`** is the canonical re-injection hook. Server-side compaction (API beta header `compact-2026-01-12`) is the underlying mechanism for Opus 4.6/4.7 and Sonnet 4.6.

**Prompt caching** mechanics deserve precision[^23]. Default TTL is 5 min (refreshed free on each hit); extended TTL is 1 h via `cache_control: {"type":"ephemeral","ttl":"1h"}`. **5-min cache write costs 1.25x base input; 1-h cache write costs 2x base input; cache read costs 0.1x base input**. Minimum cacheable prompt is 1024 tokens on Sonnet 4.5 and Opus 4-4.1, 2048 on Sonnet 4.6 and Haiku 3.5, and **4096 on Opus 4.5/4.6/4.7 and Haiku 4.5**. Maximum 4 explicit breakpoints per request (auto-cache uses one slot); 20 content-block lookback before each breakpoint. **Hierarchy and invalidation** flows `tools -> system -> messages`: modifying tools invalidates everything; toggling web search, citations, or `speed: "fast"` invalidates system plus messages; adding/removing images, changing `tool_choice`, or changing thinking parameters invalidates messages cache only. **Cache hits require 100% identical prefix** to the breakpoint. **Cache writes happen only at the breakpoint** -- placing a breakpoint on per-request varying content (timestamps, the user message) silently produces zero cache reads. **When mixing TTLs, 1-h breakpoints must precede 5-min breakpoints in the prompt order**. Claude Code automatically caches system prompt plus tool definitions plus CLAUDE.md plus conversation history; ~95-96% cache-hit rates are normal on long sessions, and Anthropic engineers reportedly declare incidents when rates drop sharply.

## 7. Custom skill authoring done well

Anthropic released **Agent Skills as an open standard on 2025-12-18** at `agentskills.io`, since adopted by OpenAI Codex CLI, Google Gemini CLI, GitHub Copilot, and Cursor[^24]. A skill is a directory whose entrypoint is `SKILL.md`, beginning with YAML frontmatter and continuing in markdown. Required fields are `name` (lowercase plus hyphens, <=64 chars, regex `^[a-z0-9]+(-[a-z0-9]+)*$`) and `description` (<=1,024 chars per the open standard; Claude Code further truncates `description + when_to_use` at **1,536 characters** in discovery). Other recognized fields: `when_to_use`, `allowed-tools`, `argument-hint`, `arguments` (named positional args usable as `$file`, `$format`), `model`, `effort`, `disable-model-invocation`, `user-invocable`, `context: fork`, `agent`, `paths` (glob auto-trigger filter), `hooks`, `shell`, plus open-standard `license`, `compatibility`, and `metadata` map. **String substitutions** in body text include `$ARGUMENTS`, `$ARGUMENTS[N]`, `$N`, `$<name>`, `${CLAUDE_SESSION_ID}`, `${CLAUDE_SKILL_DIR}`, plus **inline shell injection** via `` !`<cmd>` `` or fenced ```` ```! ```` blocks that run before the prompt is sent.

**Progressive disclosure** is the load-order discipline that keeps the harness fast[^25]. Layer one is metadata (`name` + truncated `description`), median ~80 tokens, always in the system prompt -- Anthropic's 17 official skills together total ~1,700 tokens of metadata. Layer two is the SKILL.md body, loaded only when the skill triggers; target under 500 lines and under 5,000 words. Anthropic's bodies range from ~275 tokens (`internal-comms`) to ~8,000 (`skill-creator`) with a median around 2,000. Layer three is bundled files in `references/` (read on demand), `scripts/` (executed without entering context), and `assets/` (templates, fonts, icons) -- effectively unbounded. Reference files should be linked from SKILL.md so Claude knows when to load each: *"For complete API details, see [reference.md](reference.md)."*

**Description-as-trigger** is the single highest-leverage authoring decision. Claude only sees the description at startup, so it must encode both *what the skill does* and *when to invoke it*. Best practices from Anthropic's `skill-creator` and platform docs: write in third-person, action-oriented voice ("This skill should be used when...", not "Use me to..."); front-load the use case before the 1,536-char truncation cliff; be **"pushy"** to fight under-triggering ("Make sure to use this skill whenever the user mentions dashboards, data visualization, internal metrics, or wants to display any kind of company data, even if they don't explicitly ask for a 'dashboard'"); include both the action and specific trigger phrases; use gerund-form names (`explain-code`, `pdf-processing`). Note that Claude will not trigger a skill on trivial one-step queries it can already handle regardless of description quality.

**The `allowed-tools` restriction model is widely misunderstood**. It is a *pre-approval list*, not a sandbox: it grants the listed tools permission-free use while the skill is active, but every tool remains callable; unlisted tools fall back to session permission rules. **To actually block tools while a skill runs, add deny rules in `/permissions`**. SDK note: `allowed-tools` in frontmatter is honored only by the Claude Code CLI; the Agent SDK requires `allowedTools` plus `permissionMode: "dontAsk"`.

**Mode routing within a skill** uses two patterns. Argument dispatch declares `arguments: [mode, target]` and branches body content on `$mode`; invoked as `/skill-name mode-name args` with `$ARGUMENTS[0]` resolving to `mode`. Domain organization keeps SKILL.md lean as a router and offloads variants to `references/aws.md`, `references/gcp.md`, etc., loaded on demand -- recommended by `skill-creator` for cloud-deploy or multi-framework skills. **Skill discovery locations** in precedence order are enterprise managed, personal (`~/.claude/skills/<name>/`), project (`.claude/skills/<name>/`, plus nested `packages/*/.claude/skills/`), plugin (`<plugin>/skills/<name>/` namespaced as `plugin-name:skill-name`), and additional dirs from `--add-dir`. Live change detection is on -- adding or editing within a watched directory takes effect mid-session.

**The `skill-creator` `measure-skill-performance` evaluation harness is the canonical eval methodology** and few operators run it[^26]. The flow saves 2-3 realistic prompts to `evals/evals.json`, spawns paired with-skill and baseline subagents in parallel for each case (baseline being no skill for new skills, or the prior version for improvements), drafts objectively-verifiable assertions while runs execute, captures `total_tokens` and `duration_ms` into `timing.json`, grades via `agents/grader.md`, aggregates with `python -m scripts.aggregate_benchmark` into `benchmark.json` and `benchmark.md` (mean +/- stddev, deltas, pass_rate), and runs `eval-viewer/generate_review.py` *before* self-review so the user can leave feedback exporting to `feedback.json`. Each iteration writes to a new `iteration-<N+1>/` directory with `--previous-workspace` chained. **Description optimization** uses `scripts/run_loop.py`: 5 iterations, 60/40 train/test split, 3x per query for stable trigger rate, picks `best_description` by *test* score to avoid overfitting. The eval set should mix 8-10 should-trigger and 8-10 should-not-trigger queries with negatives that are near-misses, not obviously irrelevant. Final packaging is `scripts/package_skill.py` producing a `.skill` archive.

**Common authoring mistakes** in order of frequency: vague descriptions that never trigger ("Format this data" -- should be a concrete, pushy, trigger-phrase-rich description); bloated SKILL.md that eats context (over ~500 lines means split into `references/`); missing `allowed-tools` so the skill silently fails on unapproved tools; no input->output exemplars for the model to pattern-match; tool-locking confusion (`allowed-tools` grants, doesn't restrict); over-rigid MUSTs and ALWAYSs (skill-creator guidance recommends explaining *why*); overfitting evals to test cases; and forgetting `disable-model-invocation: true` on side-effectful skills like `/deploy`, `/commit`, `/send-slack-message`.

## 8. Plan mode, background execution, worktrees, and scheduling

**Plan mode** has three entry points: Shift+Tab cycles Normal -> Auto-Accept -> Plan -> Normal (footer shows `[pause] plan mode on`); `/plan` slash command since v2.1.0; and `--permission-mode plan` for headless startup. Windows footnote: until v2.1.47 Plan was sometimes dropped from the Shift+Tab cycle on Windows -- fall back to Alt+M; v2.1.47 also fixed Plan being silently lost on auto-compaction. Tool gating allows Read, LS, Glob, Grep, Task, TodoRead/TodoWrite, WebFetch, WebSearch, NotebookRead, AskUserQuestion; forbids Edit, MultiEdit, Write, Bash, NotebookEdit, and state-modifying MCP tools. **`EnterPlanMode` and `ExitPlanMode` are real internal tools** (verified via Anthropic GitHub issues #19623, #32934, #9701 and the Piebald-AI system-prompt mirror); Claude can self-enter plan mode mid-session. `ExitPlanMode(plan: "...")` surfaces an approval dialog, and on approval the plan markdown is written to `~/.claude/plans/*.md` and the session transitions to acceptEdits regardless of prior mode (issue #39973)[^27]. The `opusplan` model alias plans with Opus 4.7/4.6 in 200K context and executes with Sonnet 4.6.

**Background bash execution** uses the Bash tool's **`run_in_background: true`** parameter, which returns immediately with a `bash_id` (e.g., `bash_1`). Ctrl+B triggers it interactively. Lifecycle: output writes to a file; Claude retrieves new output via the **`BashOutput`** tool (`{ bash_id, filter? }` returns only new output since last check plus `<status>running|completed|killed</status>` plus exit code); **`KillBash`** / `KillShell` kills by `shell_id`; **`/bashes`** lists running and completed shells with IDs, runtimes, and exit codes. Auto-cleanup on session exit, auto-terminate at 5 GB output, disable via `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1`. **The brief's `Monitor` tool is unverified in primary documentation** -- `BashOutput` is the canonical poll tool. **`TaskOutput` is referenced in user-filed issues #15095 and #29011 as the read-side tool for task IDs returned by Bash or Task with run_in_background, but is absent from official docs**; treat as semi-confirmed and prefer `BashOutput`.

**Worktree isolation** shipped natively in Claude Code v2.1.49 (Feb 19 2026) and was extended in v2.1.50[^28]. Three forms: CLI flag (`claude --worktree feature-auth` or `-w`, optionally with `--tmux`), creating `.claude/worktrees/feature-auth/` on branch `worktree-feature-auth`; subagent frontmatter (`isolation: worktree`) per invocation; and `WorktreeCreate`/`WorktreeRemove` hooks for non-git VCS bootstrap. **Add `.claude/worktrees/` to `.gitignore`** -- not done by default. Files in `.gitignore` aren't carried over; use `.worktreeinclude` to copy `.env` or `config/master.key` into each new worktree. **Cleanup semantics**: no changes -> worktree plus branch removed automatically; changes/commits exist -> user prompted to keep (preserves dir and branch for resume) or remove (deletes both, including uncommitted work); orphaned worktrees from crashes auto-remove on next startup once older than `cleanupPeriodDays` AND clean. **Worktrees beat sequential edits** when exploring mutually-exclusive approaches in parallel ("try three solutions, show me diffs"), when hot-fixing on a separate branch while a long refactor continues, or when batching migrations across many files (the `/batch` command relies on this pattern). **Resume across worktrees**: `claude --resume <name>` resolves across the repo and its worktrees; sessions started via `claude -p` or SDK don't appear in pickers but resume by session ID.

**Scheduling** runs through three primitives at different durability levels[^29]. **`/loop`** is bundled and session-scoped -- fastest path: `/loop 5m check if the deploy finished`, `/loop 20m /review-pr 1234` (slash commands nest), or `/loop` alone (added v2.1.101 -- self-paced, Claude picks its own cadence using the new **`ScheduleWakeup`** tool, also v2.1.101). Units are `s` (rounded to 1 min cron granularity), `m`, `h`, `d`. **`CronCreate` / `CronList` / `CronDelete`** back both `/loop` and natural-language one-shot reminders, accept standard 5-field cron with `*`, `5`, `*/15`, `1-5`, `1,15,30` (no `L`, `W`, `?`, or `MON`/`JAN` aliases), apply OR semantics when day-of-month and day-of-week are both constrained, and use local timezone. Limits: up to **50 scheduled tasks per session**; recurring tasks expire 7 days after creation, fire one final time, and self-delete; the scheduler polls every second, queues at low priority, fires between Claude turns and never mid-response; recurring fires up to 10% late capped at 15 min, one-shots at `:00`/`:30` fire up to 90 s early (avoid by picking minutes other than 0/30); no catch-up for missed fires; resume restores tasks within their TTLs but **background bash and monitor tasks do not survive resume**; disable via `CLAUDE_CODE_DISABLE_CRON=1`. **Durable scheduling** for cross-session work uses Desktop scheduled tasks (macOS/Windows only) or **Routines** on Anthropic-managed infrastructure or GitHub events or API trigger.

**`RemoteTrigger` is the event-side complement** to time-based scheduling, verified through Anthropic GitHub issues #43802, #42652, and #51151 plus the CHANGELOG entry "Fixed RemoteTrigger tool's `run` action sending an empty body." It calls `POST /v1/code/triggers` to create remote-scheduled or webhook-driven triggers, letting external systems (GitHub webhooks, form submits, email parsers) wake a Claude Code invocation on demand. The tool requires the account's organization UUID resolution from the OAuth token; some Max-plan setups return "Unable to resolve organization UUID." A composable orchestration recipe combines `CronCreate` (time) + `RemoteTrigger` (events) + `isolation: worktree` (filesystem) + Bash `run_in_background` plus `BashOutput` (background work) without an always-on daemon -- for example, a 5-minute cron that runs a `/heartbeat-pulse` skill that fires `RemoteTrigger` only when work exists.

## 9. Power-user patterns and anti-patterns

**Anthropic-canonical patterns** start with Boris Cherny's **Explore -> Plan -> Code -> Commit** loop[^30]: for any non-trivial PR, Shift+Tab twice into Plan mode, iterate on the spec until the plan is correct, switch to auto-accept edits, and let Claude usually one-shot the implementation. Variants are Explore->Plan->Confirm->Code->Commit and Write-Tests->Commit->Code->Iterate->Commit (TDD) and Code->Screenshot->Iterate (UI). **TDD with subagents** writes failing tests in the parent, dispatches a `code-writer` subagent to make them pass, and runs the tests in the parent to verify; Anthropic ships a Stop-hook recipe that re-fires Claude until `npm test` returns zero. Cherny calls verification loops "probably the most important thing" -- 2-3x output quality. **Parallel exploration** spawns multiple subagents in parallel and a separate set of "challenge" subagents to poke holes in the first set's findings, yielding "all real issues without the false ones"[^31]. **Security-review pipelines** chain code-reviewer -> `/review` -> `/security-review`, with the GitHub `claude-code-security-review` Action and the managed Code Review PR-comment service stacking similar coverage; `REVIEW.md` is injected at top priority into every reviewer agent. **Prompt-caching tactics** from Thariq Shihipar (Claude Code team)[^32]: never put timestamps in the system prompt -- inject via `<system-reminder>` in the next user/tool-result message; tool order must be deterministic; mid-session model switching rebuilds the cache from scratch (Opus -> Haiku for one easy question is *more* expensive than letting Opus answer); Plan mode keeps tools in-prompt and uses `EnterPlanMode`/`ExitPlanMode` *as tools* rather than swapping toolsets so the cache survives; MCP overflow handled via `defer_loading: true` stubs and a Tool Search tool. **Long-context decisions**: use 1M for single-shot analysis of large codebases or full diffs and sustained agent traces with intermediate findings; do not use 1M for routine 80-120K sessions (compaction or `/clear` is cheaper and the model reasons better with less stale context); Sonnet 4.5 MRCR at 1M was 18.5% versus Opus 4.6 at 76-78.3%. **Compaction-survival design** keeps persistent rules in CLAUDE.md not chat, uses a "Compact Instructions" section in CLAUDE.md or `/compact focus on X`, snapshots SESSION_STATE.md via PreCompact and re-injects via PostCompact, designs skills to re-load themselves from disk, and compacts proactively at ~60% rather than waiting for 95%. **Continuous reset / aggressive `/clear`** is a documented Cherny pattern -- five numbered terminals plus 5-10 web sessions, each tab a fresh session per task. Other Anthropic-sanctioned patterns: PostToolUse format hooks, Stop-hook test-gating, slash-commands in `.claude/commands/` checked into git for inner-loop tasks, and `CLAUDE_CODE_SUBAGENT_MODEL=sonnet` to route subagents to cheaper models while main runs Opus.

**Patterns from named community users**. **Simon Willison** practices README-driven development -- construct the README describing exactly how the tool should work, then dump it into Claude Code with red/green TDD -- and publishes transcripts via his `claude-code-transcripts` tool (Dec 25 2025) for review and audit, with `scan-for-secrets` vetting before publication[^33]. **Armin Ronacher** runs Sonnet exclusively on the $100 Max plan, optimizes for token efficiency, avoids screenshots and browsers, uses MCP only when alternatives are too unreliable (e.g., Playwright MCP), and otherwise lets Claude drive native CLIs (`psql`); he manages cache breakpoints explicitly and recommends Go for new backend agentic projects[^34]. **Geoffrey Huntley's "Ralph"** pattern (`while true; do cat PROMPT.md | claude; done`) is now shipped as the official `ralph-wiggum` plugin in `anthropics/claude-code` using a Stop hook with `--completion-promise`; Huntley's key rules are not allocating to the primary context, spawning parallel subagents for expensive allocation, and keeping `fix_plan.md` and `specs/` deterministically on the stack each iteration[^35]. **Thorsten Ball** frames agents as a button that produces 500 lines of glue code in a minute and insists operators must understand all code the agent writes; "doom loops" are real. **Harper Reed** runs three-phase brainstorm-spec -> plan -> execute with `spec.md` -> `prompt_plan.md` -> `todo.md` (the LLM checks items off as state) and uses pre-commit hooks (lint, format, type) as the verification loop because "the robot really wants to commit"[^36]. **Steve Krouse** (Val Town) builds platform-opinionated system prompts rather than relying on user-side rules.

**Anti-patterns** worth internalizing because each silently degrades sessions:

- **Over-skilling**: a skill for every micro-task hurts discoverability and routing accuracy because Claude reads every description at session start. Use `disable-model-invocation: true` for manually-invoked skills.
- **Hook cascades and infinite loops**: a PostToolUse hook that calls `claude` re-inherits all hooks (issues #10205, #3573); Stop hooks emitting `{"continue": true}` or system-message "next steps" trigger Claude to keep working. Always check `stop_hook_active`; pass `--settings no-hooks.json` with `"disableAllHooks": true` to spawned subagents.
- **Permission escalation drift**: starting in default, hitting friction, and finishing in `bypassPermissions` for the rest of the session. Cherny does not use `--dangerously-skip-permissions`; instead pre-approves safe commands in settings.json. CVE-2025-55284 (API key DNS exfiltration) and the IDEsaster disclosure cite this drift as the highest-impact local risk.
- **Context bloat in CLAUDE.md**: 800+ line files. Cherny's CLAUDE.md is ~100 lines; the AGENTS.md community standard recommends a sub-300 line ceiling and over-500 means "you are fighting your config." Add a line only when Claude makes a mistake worth preventing; remove when the underlying issue resolves.
- **Subagent over-delegation**: the parent dispatches everything, loses thread, and cannot verify. Symmetric anti-pattern: subagent given too narrow a context (the Agent tool prompt is the only inbound channel) so it cannot reason about callers. "Most sub-agent failures aren't execution failures -- they're invocation failures."
- **MCP proliferation**: 12 active servers measured at 82K of 200K tokens (41%) of context burn before any conversation; mcp-omnisearch alone consumed 14,114 tokens. Each server is a subprocess, and >20 measurably slows startup. Use Tool Search / `defer_loading: true` stubs, McPick-style toggle tools, and project-scoped `.mcp.json` over global.
- **CLAUDE.md sprawl**: every subdirectory with its own conflicting CLAUDE.md, silently competing because subdir rules are scoped to that path.
- **Memory pollution**: auto-saving everything as memory crowds the cached prefix; the first 200 lines / 25 KB of MEMORY.md load every session. Curate aggressively.
- **Compaction-induced amnesia**: critical state lives only in chat transcript and gets lossily summarized at 95%. Fix with PreCompact persisting SESSION_STATE.md, "Compact Instructions" in CLAUDE.md, or `/compact focus on X`.
- **Unverified subagent outputs**: parent trusts the subagent summary because intermediate tool calls are hidden. Cherny's mitigation: spawn a second set of "critic" subagents that contradicts the first set's findings; net ~30% more real issues caught.
- **Cache-breaking habits**: today's date in the system prompt; non-deterministic tool ordering; mid-session model switches; toggling MCP tools per request. Each invalidates the entire prefix. Indicator: `cache_creation_input_tokens` and `cache_read_input_tokens` both zero (silent failure).
- **1M-context misuse**: running routine 80K sessions on Opus 4.6 1M just because it is enabled (no harm but no help), or using Sonnet 4.5 at 1M for retrieval (18.5% MRCR). `/clear` plus a fresh session is usually better.
- **Doom loops without specs**: running autonomous Ralph loops without a near-perfect spec; the model chases its compile-clean reward function and produces placeholder implementations. Huntley's mitigation: explicit "DO NOT IMPLEMENT PLACEHOLDER OR SIMPLE IMPLEMENTATIONS" in `PROMPT.md`.
- **PR-quality collapse / "slopforks"** (Ronacher, Feb 13 2026, Mar 5 2026): writing speed outruns review capacity; OpenClaw at 2,500+ open PRs. Velocity is not throughput when reviewer time is the queue.
- **Persona drift / parasocial usage** (Ronacher, "Agent Psychosis," Jan 18 2026): always-on memory agents create unhealthy reinforcement loops, especially among non-engineers, and AI-generated PRs that look correct but aren't impose maintainer overload.

A concrete to-do list for the operator: try `/ultrareview` on the next substantial branch (3 free runs expire May 5 2026); add a PreCompact + PostCompact hook pair for SESSION_STATE.md snapshotting; pilot worktree isolation via `claude --worktree` for parallel exploration; replace one path-guard PreToolUse rule with a deny-rule in `permissions.deny` for stronger enforcement; run `skill-creator`'s `measure-skill-performance` harness against two existing skills; convert one always-loaded MCP server to `defer_loading: true`; add `/loop` plus `ScheduleWakeup` to a long-running monitoring task; replace any `--dangerously-skip-permissions` habit with a curated `permissions.allow` list; spawn `/simplify` on a recent PR before merge; set `CLAUDE_CODE_SUBAGENT_MODEL=sonnet` and route Opus to the main thread only.
